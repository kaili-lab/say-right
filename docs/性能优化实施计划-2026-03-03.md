# 性能优化实施计划

> 基线文档：`docs/性能分析综合报告-2026-03-03.md`
> 目标：修复首页/复习/卡片组 Tab 的慢加载问题（4-5s → <1s）。

## 1. 范围与非范围

- 范围：`backend/` 读路径、鉴权路径、首页聚合查询、性能观测。
- 非范围：不做 async 全量重构；不做前端 UI 重构；不改业务契约语义。

## 2. 分阶段实施

## Phase 0（低风险高收益，改动量小）

### P0-1 删除连接池出池探活

- 文件：`backend/app/db/pool.py`
- 动作：删除 `check=ConnectionPool.check_connection` 参数。现有 `_run_with_retry` 已能处理连接被服务端回收的情况，health check 纯属多余。
- 预期：每次仓储调用减少 1 次 `SELECT 1` 网络往返。

```python
# 改动后
pool = ConnectionPool(
    conninfo=database_url,
    min_size=min_size,
    max_size=max_size,
    open=True,
)
```

### P0-1.1 瞬时连接重试扩展到所有 Postgres 仓储

- 文件：`backend/app/card/repository.py`、`backend/app/review/repository.py`
- 现状：只有 `auth` 和 `deck` 的 repository 有 `_run_with_retry`，`card`（8 个方法）和 `review`（7 个方法）全是裸调 `self._pool.connection()`，碰到过期连接直接抛异常导致 503。
- 动作：将 `_run_with_retry` 提取为公共辅助（如 `backend/app/db/helpers.py`），`card` 和 `review` 的 Postgres 仓储统一接入。
- 必要性：删掉 health check 后，连接池中可能存在被 Neon 服务端回收的空闲连接，没有 retry 的路径会偶发 503。

### P0-2 读操作使用 autocommit

- 文件：`backend/app/auth/repository.py`、`backend/app/deck/repository.py`、`backend/app/card/repository.py`、`backend/app/review/repository.py`
- 动作：只读方法取连接后设置 `conn.autocommit = True`，省去事务收尾的 COMMIT 往返。写方法保持原事务语义不变。无需手动恢复 — `psycopg_pool` 归还连接时自动 reset。

```python
# 读方法示例
with self._pool.connection() as conn:
    conn.autocommit = True
    with conn.cursor(row_factory=dict_row) as cur:
        cur.execute(sql, params)
        return cur.fetchone()
```

### P0-3 移除稳态冗余 ensure_default_deck

- 文件：`backend/app/deck/service.py`
- 动作：`list_decks` 改为"先 list；为空再 ensure + 重查"。用户注册时已通过 `on_user_registered` 回调创建默认组，稳态下不会走到 ensure 分支。
- 预期：`/decks`、`/review/decks` 各少 1 次查询。

```python
def list_decks(self, user_id: str) -> list[Deck]:
    decks = self.repository.list_by_user(user_id)
    if not decks:
        self.repository.ensure_default_deck(user_id)
        decks = self.repository.list_by_user(user_id)
    return decks
```

### P0-4 去掉首页重复查 user

- 文件：`backend/app/dashboard/api.py`、`backend/app/dashboard/service.py`
- 动作：路由层将鉴权得到的 `current_user` 直接传入 service，service 不再调用 `user_repository.get_by_id()`。`DashboardService` 移除对 `user_repository` 的依赖。
- 预期：首页少 1 次查询。

### P0-5 鉴权结果短 TTL 缓存

- 文件：`backend/app/auth/service.py`
- 动作：在 `get_current_user` 中对 `user_repository.get_by_id()` 结果加 TTL 60s 内存缓存，仅缓存查到用户的情况。

```python
from threading import Lock
from cachetools import TTLCache

_user_cache: TTLCache[str, User] = TTLCache(maxsize=256, ttl=60)
_user_cache_lock = Lock()

# 在 get_current_user 中：
with _user_cache_lock:
    user = _user_cache.get(payload.sub)
if user is None:
    user = self.user_repository.get_by_id(payload.sub)
    if user is not None:
        with _user_cache_lock:
            _user_cache[payload.sub] = user
```

- 线程安全：同步路由在 FastAPI 线程池中并发执行，`TTLCache` 内部的过期清理 + 插入在并发下可能损坏数据结构，需用 Lock 保护。

- 安全边界：TTL 60s 内用户密码变更不影响已签发 token 的有效性（token 本身有独立过期时间）；登录/注册路径不经过此缓存。

### P0-6 请求耗时日志

- 文件：`backend/app/main.py`
- 动作：添加计时中间件，记录每个请求的耗时。

```python
@app.middleware("http")
async def timing_middleware(request, call_next):
    start = time.perf_counter()
    response = await call_next(request)
    ms = (time.perf_counter() - start) * 1000
    logger.info("%s %s → %.0fms", request.method, request.url.path, ms)
    return response
```

- 文件：`backend/app/deck/repository.py`、`backend/app/review/service.py`

## 4. 验收标准

> **环境声明**：以下绝对阈值基于本地开发环境到 Neon ap-southeast-1 的网络条件（单次 RTT ~150-300ms）。若网络环境变化，以相对基线提升 >= 60% 作为辅助判断依据。

真实 Postgres 环境、同网络条件、20 次采样（排除预热）：

- `/decks` P95 <= 500ms
- `/review/decks` P95 <= 500ms
- `/dashboard/home-summary` P95 <= 800ms（含 5000+ logs 用户）
- 5xx = 0（数据库瞬时故障映射为 503）
- 现有功能测试全部通过

## 5. 风险与回滚

| 风险 | 对策 |
|------|------|
| autocommit 误用于写路径 | 只在读方法中设置；写方法不改动 |
| 鉴权缓存导致短时旧数据 | TTL 60s；仅缓存正向命中；token 有独立过期机制 |
| 聚合 SQL 可读性下降 | 单独 repository 文件 + SQL 注释 + EXPLAIN 结果留档 |

回滚策略：Phase 0 和 Phase 1 分开提交，任一阶段可独立回滚。

## 6. 执行顺序

1. Phase 0 全部完成 → 复测，预期 `/decks` 和 `/review/decks` 降到 ~500ms，首页降到 ~1.5-2s。
2. Phase 1 聚合改造 → 复测，预期首页降到 <800ms。
3. 若仍不达标，进入 Phase 2。
