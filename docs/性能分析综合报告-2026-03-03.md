# 性能分析综合报告（合并版）

> 基于两份文档合并：
> - `docs/性能分析报告.md`（Claude）
> - `docs/性能诊断-Tab加载慢-2026-03-03.md`（本地实测）

## 1. 最终结论

- **根因不是 FastAPI 框架慢**，而是当前请求链路里数据库往返次数过多，且每次 RTT 成本高。
- **连接池应保留**（能显著降低重复建连成本），但当前配置与查询组织方式仍导致“池化后依旧慢”。
- 当前 4-5 秒属于“串行多查询 × 跨区网络延迟 × 冗余探活/事务开销”的叠加结果。

## 2. 双方共识（已确认）

1. 单请求存在 3~6 次串行数据库调用，RTT 叠加明显。  
2. `check=ConnectionPool.check_connection` 会给每次出池增加额外往返开销（`backend/app/db/pool.py:18`）。  
3. 首页链路存在冗余查询：鉴权查用户后，首页服务又查一次用户（`backend/app/dashboard/service.py:60`）。  
4. `DeckService.list_decks()` 稳态仍先 `ensure_default_deck`，有额外查询（`backend/app/deck/service.py:25`）。  
5. 首页目前把 `review_logs` 拉全量到 Python 聚合，数据量增长会持续退化（`backend/app/dashboard/service.py:59`）。

## 3. 合并后校正（对原结论的修订）

### 3.1 连接池不是“无效优化”

实测表明：
- 无池（每次建连）`connect + query` 约 1.1s
- 有池后续查询约 0.35s

结论：**连接池不是慢的根因，且有必要保留**。

### 3.2 `channel_binding` 影响存在但非主因

实测：
- 含 `channel_binding` 平均约 1144ms
- 去掉后约 1102ms

有改善，但量级远小于“减少查询次数”。因此属于 **次优先级优化**。

### 3.3 `async def` 不是首要解法

同步路由本身不是当前瓶颈。当前优先级应是：
1) 减少查询次数；2) 降低每次查询往返；3) 再评估 async 改造。

## 4. 根因模型（统一版）

- **A（高）**：接口查询拆分过细、串行执行，RTT 叠加。  
- **B（高）**：每次出池健康检查增加一次额外往返。  
- **C（高）**：读请求默认事务模式带来额外收尾成本。  
- **D（中）**：默认组 ensure 在稳态重复执行。  
- **E（中-高）**：首页全量日志回传后 Python 聚合，数据增长时劣化明显。  
- **F（中）**：鉴权查询与业务查询存在重复读取 users。

## 5. 统一解决方案（分阶段）

## Phase 0（立即执行，低风险）

1. 连接池保留，但移除默认 `check_connection`（改为可配置开关）。  
2. `list_decks` 改为“先 list；为空再 ensure + 重查”。  
3. 首页复用鉴权用户，去掉第二次 `get_by_id`。  
4. 增加请求耗时与仓储耗时日志（便于回归比对）。

## Phase 1（核心收益）

1. 将首页统计改为 SQL 聚合，禁止全量 `review_logs` 回传应用层。  
2. 新增/校准索引支撑“每卡最新评分”聚合查询。  
3. 对 `/decks`、`/review/decks` 做查询合并，减少往返数。

## Phase 2（可选增强）

1. 评估短 TTL 缓存（首页 30~60s）。  
2. 评估 `channel_binding` 调整（作为次级优化）。  
3. 如仍不达标，再评估 async 路由与并发查询改造。  
4. 优化部署拓扑（应用与数据库同区域/近区域）。

## 6. 验收标准（统一）

真实 Postgres 环境，连续 20 次采样：
- `/decks` P95 <= 500ms
- `/review/decks` P95 <= 500ms
- `/dashboard/home-summary` P95 <= 800ms（含 5000+ logs 用户）
- 5xx = 0（数据库瞬时故障应返回 503）

## 7. 为什么 Hono 体感更丝滑（统一解释）

- 你原有 Hono + Drizzle + Neon serverless driver 属于 HTTP/Serverless 访问路径，很多连接复用细节由平台侧处理。  
- 现在 FastAPI + psycopg 走传统 PG 连接路径，连接、事务、查询组织都在应用侧显式承担。  
- 因此差异不在“框架标签”，而在 **访问协议、部署拓扑、查询设计**。

## 8. 下一次实施范围（建议）

- 仅落地 Phase 0 + Phase 1；不在本轮做 async 大改。  
- 每一项变更都输出改造前后采样对比（平均值 + P95）。  
- 将性能回归测试纳入 CI（至少夜间任务跑真实 Postgres 基准）。

## 9. 补充遗漏（Claude 审阅）

### 9.1 读操作 autocommit 应纳入 Phase 0

根因 C 已标注为**高影响**，实测数据显示 autocommit 读从 357ms 降到 149ms（单次查询减半）。但 Phase 0 执行清单中未包含此项。

**建议**：在 Phase 0 增加一条——仓储读方法使用 `connection.autocommit = True` 或 `pool.connection(autocommit=True)`，避免只读查询的事务收尾往返。改动量小，收益明确。

### 9.2 auth 鉴权结果应加短 TTL 缓存

报告已解决"首页内部重复查 user"（根因 F），但 **auth 依赖本身每次请求都查数据库** 的问题未覆盖。`get_current_user()`（`backend/app/auth/service.py:83`）每次请求都执行 `user_repository.get_by_id()`，产生一次 ~300ms 的数据库往返。

**建议**：对 `get_by_id` 结果加短 TTL 内存缓存（如 `cachetools.TTLCache`，60s 过期），同一用户连续请求可免去此开销。归入 Phase 0 或 Phase 1。
