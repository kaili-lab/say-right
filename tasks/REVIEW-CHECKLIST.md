# Task Review Checklist（面向非 Python/FastAPI 背景）

> 用途：每完成一个 task，按此清单做快速验收（10~15 分钟）。

## 1) 先看任务边界（防止做偏）

- 当前 task 的 `目标`
- `范围` 与 `不在范围`
- `DoD` 与 `test_commands`

## 2) 先黑盒验证（优先）

- 先看集成测试文件（接口流程与边界行为）
- 执行 task 指定命令，确认通过

## 3) 再白盒抽查（结构是否清晰）

- 路由层（API/Controller）：请求解析、状态码映射
- 服务层（Service）：业务规则与流程编排
- 仓储层（Repository）：数据读写与唯一性约束
- 领域模型（Domain）：核心数据结构与不变量

> 建议先判断“分层职责是否清晰”，再看语法细节。

## 4) 必看交付证据

- 变更文件清单（带行号）
- 测试命令 + 退出码 + 关键通过行
- 任务收尾记录：已完成 review，且存在本 task 对应的 `commit + push` 证据（commit id / 远端分支）
- 状态文件是否更新：
  - `tasks/*/INDEX.md`
  - `tasks/*/HANDOFF.md`
  - `tasks/*/SESSION-MEMORY.md`（若该目录存在）
  - 当前 task 的 `output_summary`

## 5) 快速风险检查

- 是否超出当前 task 范围
- 是否引入未声明依赖
- 是否遗漏边界用例
- 是否出现“测试过度 mock”
- 若存在上游未就绪：是否显式声明 `test_data_strategy`，并说明后续回归替换点
- 是否出现“强行通过测试”（弱化断言/跳过用例/删除失败用例）
- 若目录有 `SESSION-MEMORY.md`：是否追加了本任务的经验/踩坑记录
- 是否出现“未 commit/push 就开始下一个 task”的流程违规
