# 项目级 AGENTS 增补规则

本文件是项目级增补规则，与系统/上级规则共同生效，不替代更高优先级指令。

## 单一事实来源（请优先参考）

- 任务模板、字段定义、执行流程：`tasks/任务拆分说明-final.md`
- 任务验收清单与审阅包模板：`tasks/REVIEW-CHECKLIST.md`

## 后端任务必读

- 执行 `backend/` 相关 task 前，必须先阅读：`docs/FastAPI项目固定流程.md`。
- 执行 `backend/` 相关 task 完成前，必须执行并通过：`make -C backend check`。

## 语言与沟通

- 所有输出使用中文。

## 版本策略

- 默认使用各技术栈的最新稳定版（GA），禁止无理由沿用过时主版本（如在无约束下继续使用 React 18）。
- 若受契约、上游依赖或兼容性约束必须锁定旧版本，需在任务输出中明确说明原因、影响范围与后续升级建议。
- 新增依赖时，优先选择当前生态主流且维护活跃的稳定版本，避免实验版/预发布版进入主分支。

## TDD 与测试红线（强制）

- 前后端任务默认采用 TDD：先写测试（Red）-> 最小实现（Green）-> 重构（Refactor）。
- 每个任务必须声明并执行 `test_commands`，保留可追溯证据（命令、退出码、关键通过行）。
- LLM 相关测试必须使用可复现的 stub/fixture，禁止依赖真实模型调用。
- 若前置模块未就绪，必须在 task 中显式声明 `test_data_strategy`，并在 `output_summary` 标注后续替换点。
- 除“前置模块未就绪且已声明策略”外，任何测试异常都不得通过弱化断言、跳过用例、滥用 mock、删除用例等方式“强行通过”。
- 出现测试异常时，必须先向用户报告失败现象、根因判断、可选修复方案与影响范围，等待确认后继续。

## 学习型注释与文档策略

- 新增/修改文件时，默认补充文件级说明（模块 docstring）与对外类/方法 docstring（WHAT + WHY）。
- 行内注释遵循“关键逻辑才写”，避免逐行翻译代码。
- 复杂业务规则、异常映射、兼容性处理必须写中文注释解释原因。

## 任务交付审阅包（强制）

- 每个 task 完成后，必须提供“可复查审阅包”。
- 至少包含：契约对照、变更文件清单（带行号）、测试证据、范围核对、已知风险与下一步建议。
- 若任务失败或中断，也必须提供“失败审阅包”（失败点、原因、建议修复方向）。

## Subagent 与 Skill 规则

- 可使用 Subagent 处理可并行且边界清晰的子任务；主上下文必须完成最终复核与 DoD 收口。
- 未经主上下文复核的 Subagent 结果，不可视为任务完成。
- 任务或用户显式写出 `$SkillName` 时，必须优先使用对应 Skill。
- 任务文档存在 `skill_required` 字段时，先读取并执行对应 `SKILL.md`。
- 若请求的 Skill 不存在或不可读，需明确说明并切换到常规实现流程。
