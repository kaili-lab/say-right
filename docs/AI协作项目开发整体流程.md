# AI 协作项目开发整体流程（从问题定义到开发完成）

> 版本：v1.4  
> 整理日期：2026-02-26  
> 适用对象：使用 AI（如 Codex/Claude）协作开发的产品项目

---

## 一、文档目标

用于指导后续项目的完整开发流程，覆盖从问题定义、方案成型、任务拆解、开发、验收到复盘沉淀的全链路，减少反复改需求和反复改 UI 的返工。

---

## 二、核心原则（AI 时代）

1. **定义不清，AI 只会更快做错**  
2. **验收不严，AI 只会更快引入问题**  
3. **定义未冻结，不开工；验收未通过，不合并**

---

## 三、步骤权重（重点不平均）

- **P0（最高优先）**：步骤 1-2（定义/方案）+ 步骤 7-8（验收/复盘）
- **P1**：步骤 4-5（任务规划/接口契约）
- **P2**：步骤 6（实现执行，AI 可高效推进）

> 建议投入占比（可按项目调整）：  
> 1-2 步 40%~55%，4-5 步 15%~25%，6 步 20%~30%，7-8 步 10%~20%

---

## 四、标准流程（8 步）

## 1) 问题定义（PRD）

**做什么**
- 明确目标用户、核心场景、成功指标、非目标（不做什么）
- 澄清关键业务规则与术语定义

**产物**
- `PRD`（聚焦“做什么/为什么”）

**意义**
- 避免“开发中才发现目标不清、名词歧义”

**通过标准**
- 核心术语无歧义；主路径明确；边界条件可说明

**如何实现（推荐）**
- 先由你一次性输入尽可能完整的背景信息，再让 AI 做结构化整理与追问
- 背景信息建议至少覆盖：目标用户、使用场景、当前痛点、竞品参考、必须做/暂不做、时间与资源约束
- 采用“多轮头脑风暴 + 收敛”：
  1. 发散：AI 先给候选方案、风险点、边界条件  
  2. 收敛：你逐条确认，AI 回写为 PRD 条款  
  3. 定稿：AI 输出可评审版本（含待确认项清单）
- 每轮讨论结束都更新文档，不把关键决策只留在对话里

**AI 提问建议（用于提高需求质量）**
- “这个规则的反例是什么？”  
- “新用户首次进入时怎么走主路径？”  
- “删除/编辑/失败/空状态分别怎么处理？”  
- “哪些是 v1 必做，哪些明确不做？”

**背景信息模板（复制填空给 AI）**

```md
## 0. 项目一句话
- 我在做什么：
- 目标用户是谁：

## 1. 核心场景（3-5 条）
- 场景 1：
- 场景 2：

## 2. 当前痛点/动机
- 为什么现在要做：
- 现有方案哪里不满足：

## 3. 成功标准（可量化优先）
- 例如：完成一次核心流程的时间 < X 秒
- 例如：新用户 5 分钟内能完成首个闭环

## 4. 非目标（明确不做）
- v1 不做：
- 以后可能做：

## 5. 业务规则与边界条件（要点式）
- 数据归属/权限：
- 编辑/删除策略：
- 空状态策略：
- 错误与失败重试：

## 6. 约束（很重要）
- 平台/端：
- 技术栈偏好/限制：
- 时间/人力：
- 依赖的第三方：

## 7. 竞品/参考（可选）
- 参考产品/链接：
- 我想借鉴的点：

## 8. 名词表（避免歧义）
- 名词 A：
- 名词 B：

## 9. 开放问题（你也不确定的点）
- 问题 1：
- 问题 2：

## 10. 请你输出（给 AI 的指令）
- 产出一份结构化 PRD（含主流程、状态矩阵、边界条件）
- 列出待确认问题清单（按优先级排序）
- 列出主要风险与建议（技术/产品/体验）
```

---

## 2) 方案成型（产品方案 + 技术可行性 + Mock）

**做什么**
- 产品：页面结构、信息架构、主流程、空状态/异常态
- 视觉：主题、颜色、排版、组件规范
- 技术：关键选型与风险验证（只做必要可行性，不做过早细化）
- 用 Mock UI 反向校准 PRD（发现冲突立即修订）

**产物**
- `UI 设计规范`
- `Mock UI`
- `更新后的 PRD`
- （可选）`技术方案简版`

**意义**
- 在低成本阶段发现问题，减少代码阶段返工

**通过标准**
- 需求与 UI 相互一致；关键状态均有视觉表达

**如何实现（推荐）**
- 先出可点击的 HTML Mock（不是直接上 React），快速验证主题、颜色、布局、信息层级
- Mock UI 默认采用“多页面（多 HTML 文件）”组织方式：按最终路由/Tab 一页一个文件，便于与后续 React 页面/路由 1:1 对齐，也更利于截图基线与 E2E
- 为降低多页面维护成本，抽取共享资源：design tokens/通用样式 → `shared.css`；通用交互（如头像菜单）→ `shared.js`；每页只保留页面结构 + mock 数据
- Mock 评审按状态走查，而不是只看正常态：正常态、空状态、弹窗态、禁用态、错误态、移动端
- 采用版本迭代（如 V1/V2/V3）：每轮只解决一批明确问题，避免一次改太多
- 用 UI 反向校准需求：如果页面上表达不清，回到 PRD 补规则，不在实现阶段临时拍脑袋
- 评审结论必须落文档：哪些已确认、哪些暂缓、哪些待验证

**Mock 阶段建议产物**
- 页面清单 + 状态清单（每页至少列出关键状态）
- Mock 文件结构约定（多 HTML 页面 + shared.css/shared.js）
- 设计 token（颜色/圆角/阴影/间距/宽度）  
- 交互说明（点击后去哪里、弹窗如何关闭、按钮何时禁用）

---

## 3) 闸门冻结（G1/G2）

**做什么**
- G1：冻结核心需求（业务规则、主路径、关键术语）
- G2：冻结视觉基线（主题、颜色、布局、响应式、关键状态）

**产物**
- 冻结版本号（如 PRD v1.x / UI v1.x）

**意义**
- 给开发提供稳定基线，避免边做边改主干方向

**注意**
- 冻结不是永不修改；变更必须记录原因、影响范围、同步任务

---

## 4) 规划拆解（UI/API Tasks）

**做什么**
- 前后端拆分为小 task（单一目标、可独立验收、可追踪依赖）
- 在 task 中明确测试范围（单元/集成）与接口契约版本

**产物**
- `tasks/ui-tasks/*`
- `tasks/api-tasks/*`
- 状态文件：`INDEX.md`、`DECISIONS.md`、`HANDOFF.md`

**意义**
- 支持多会话协作，避免上下文丢失和重复劳动

**通过标准**
- 每个 task 都有目标、前置依赖、DoD、验证步骤
- 每个 task 都有 `test_scope`（unit / integration）与通过门槛

**跨任务联动标记（UI/API 分开但可协同）**
- 可以分开定义、交替开发；但必须在 task 内标注联动关系
- 统一增加字段：`paired_with`（关联任务 ID）、`contract_version`（契约版本）、`sync_point`（联调节点）
- 典型场景：`ui-task` 依赖 `api-task` 的接口返回结构，双方在同一 `sync_point` 完成联调
- 联动任务验收时，除各自 DoD 外，增加一次联合验收（最小 E2E）

---

## 5) 接口契约先行（前后端并行前提）

**做什么**
- 定义 API 路由、请求参数、响应结构、错误码、鉴权规则

**产物**
- OpenAPI / 接口契约文档 + 前端类型定义

**意义**
- 让前后端并行开发，减少联调扯皮

**通过标准**
- 契约可被前后端独立实现并通过契约校验

---

## 6) 按任务迭代开发

**做什么**
- 小步开发、小范围提交、持续集成
- 每完成一个 task 即更新状态文件
- 按 TDD 执行：先写测试（红）→ 最小实现（绿）→ 重构（Refactor）

**产物**
- 可运行增量功能 + task 完成记录

**意义**
- 降低回滚成本，问题定位更快

**TDD 执行要求（前后端）**
- 后端：业务规则优先单元测试；API + DB + 编排流程做集成测试
- 前端：组件行为优先单元测试；页面流程与状态流做集成测试
- LLM 相关测试使用可复现 stub/fixture，不直接调用真实模型
- 未先写测试的实现视为未完成，不进入合并

---

## 7) 自动化验收（必须）

**做什么（最小可行集）**
- 静态检查：lint / typecheck
- 单元测试：关键函数与业务规则
- 契约校验：API schema 兼容性
- E2E：关键主路径
- 视觉回归：固定视口截图比对（Desktop + iPhone 13）

**测试矩阵（建议）**
- 前端单元：按钮状态、表单校验、弹窗开关、文案状态切换
- 前端集成：记录页“保存触发 Agent → 结果提示 → 手动调整”完整链路
- 后端单元：分组命中、自动建组、默认组兜底、权限边界
- 后端集成：API + 数据库 + Agent 编排（stub LLM）端到端可复现

**产物**
- 测试报告 + 截图对比结果 + 缺陷清单

**意义**
- 从“主观可用”变成“客观可复现”

**通过标准（DoD）**
- 自动化检查全部通过；关键路径无阻断缺陷；视觉差异在阈值内

---

## 8) 总结复盘（资产化）

**做什么**
- 记录本次有效做法、问题根因、下次前置项
- 回灌模板与流程文档

**产物**
- 复盘文档 + 更新后的流程模板

**意义**
- 每个项目都让下一次更快、更稳

---

## 五、闸门检查清单（建议）

- **G1（需求冻结）**：核心规则、关键名词、主路径、边界条件是否确认
- **G2（视觉冻结）**：主题、布局、响应式、空状态、弹窗态是否确认
- **G3（任务冻结）**：task 颗粒度、DoD、依赖关系、接口契约是否确认
- **G4（发布准入）**：自动化通过、关键缺陷关闭、文档同步完成

---

## 六、多会话协作约定（建议强制）

每次新会话开始前，先阅读：
1. `tasks/*/INDEX.md`
2. `tasks/*/DECISIONS.md`
3. `tasks/*/HANDOFF.md`

每次完成 task 后，至少更新：
1. 当前 task 文件
2. `INDEX.md`
3. `HANDOFF.md`

---

## 七、何时允许进入开发

仅当以下条件同时满足：
- PRD 与 UI 规范已经过评审并冻结
- UI/API tasks 已拆分且具备 DoD
- 接口契约可被前后端并行实现
- 自动化验收方案已定义

---

## 八、结论

在 AI 协作开发中，真正决定效率和质量的不是“写代码速度”，而是：
- 前期定义质量
- 中期任务与契约清晰度
- 后期验收与复盘纪律

这套流程的目标是：**把反复改需求、反复改 UI 的成本前移并可控化**。

## 九、个人从零执行附录（与现有模板合并版）

> 说明：本附录只补充“个人项目高频执行动作”，不重复任务字段定义。  
> task 字段与审阅包格式以 `tasks/任务拆分说明-final.md`、`tasks/REVIEW-CHECKLIST.md` 为唯一准入标准。

### 1) 每日执行节奏（个人开发推荐）

- 上午：只做 1~2 个高价值 task 的拆分与验收标准确认
- 下午：按 task 驱动 AI 实施（Red → Green → Refactor）
- 收工前：更新 `INDEX.md` / `DECISIONS.md` / `HANDOFF.md`

建议约束：
- 同时进行的 in-progress task ≤ 2
- 当日未过门禁的改动不合并主分支

### 2) 单 task 执行口令（可直接复用）

```text
请严格按当前 task 执行：
1) 仅实现 in_scope，禁止扩展到 out_of_scope
2) 先补失败测试（Red），再做最小实现（Green）
3) 运行 task 指定 test_commands，并补充必要的全量回归命令
4) 输出审阅包：契约对照、变更文件+行号、测试证据、风险与后续
5) 若出现架构不确定项，先给方案 A/B 与影响，等待确认
```

### 3) 个人项目最小门禁（CI 建议）

- 静态检查：lint + typecheck
- 测试检查：unit + integration（至少覆盖主链路）
- 契约检查：请求/响应 schema 一致性
- 安全检查：依赖漏洞扫描（可每日定时）

判定规则：任一失败即阻断合并。

### 4) 每周质量指标（用于判断流程是否有效）

- Task 一次通过率（首次 review 即通过）
- 回归缺陷数（合并后发现）
- 平均 task 周期（创建到 done）
- AI 改动回滚率（被撤销 commit 比例）

建议目标（个人项目起步值）：
- 一次通过率 ≥ 60%
- 回归缺陷每周 ≤ 2
- 单 task 周期 ≤ 1 天
