## 一、产品概述

**Say Right** 是一款帮助用户将中文表达转化为地道英文，并通过间隔重复算法（SRS）持续复习的学习工具。

**核心价值主张**：填补市场空白——目前无任何主流 App 完整实现"AI 生成英文表达 → 保存为闪卡 → SRS 复习"的完整闭环。

**目标用户**：有英语表达需求、希望系统学习地道口语的中文用户

**平台**：响应式 Web App（HTML），手机端和桌面端均需良好支持

**用户规模**：公开产品，暂无商用计划

## 三、功能模块

### 3.1 记录（创建卡片）

**流程**：

1. 用户在输入框输入中文内容（上限 200 字）
2. 点击"生成英文"按钮，AI 根据预设提示词生成地道英文表达
3. 用户看到生成结果，可编辑 AI 给出的英文
4. 用户点击“保存卡片”后，系统先触发 Group Agent，而不是直接调用 create
5. Group Agent 完成分组并创建卡片；保存后展示结果提示，用户可立即改到其他组

**保存动作（Agent 优先）**：
- 保存入口统一由 Group Agent 编排，不走“前端直接创建记录”
- Agent 先读取已有分组并判断当前记录应归属的分组
- 若命中已有分组：Agent 直接绑定该分组并调用 create card
- 若无合适分组：Agent 先调用 create category，再拿到 category_id 调用 create card
- 若 Agent 或创建分组过程失败：回退到默认组兜底，并提示用户可手动调整
- 手动调整分组仍保留，作为 Agent 结果的人工覆盖机制

**卡片数据结构**：
- 正面：用户输入的中文
- 背面：AI 生成的英文（可编辑）
- `source_lang`：源语言（v1 固定 `zh`，为后续多语言扩展预留）
- `target_lang`：目标语言（v1 固定 `en`）
- 所属卡片组
- 创建时间
- SRS 复习状态（下次复习时间、当前间隔、易度因子等）
- 复习历史（每次复习的时间、评级来源〔AI/手动〕、最终评级）

**后期功能**：支持自定义提示词以获取不同风格的英文表达（需审核）

### 3.2 卡片组（Deck）管理

- 必须存在一个**默认组**（兜底，Group Agent 不确定时使用）
- 用户可手动创建自定义卡片组
- 每个卡片组显示三类数量：新 / 学习中 / 待复习（参考 Anki 标准顺序与配色）
- 管理页面支持查看所有组及其卡片列表
- 支持手动将卡片移动到其他组
- 默认组不可删除
- 非默认组只有在“组内无卡片”时才可删除（有卡片则禁止删除，提示先清空或迁移）

**Group Agent 逻辑**：
- 每次点击“保存卡片”时，由 Agent 执行分组与创建流程
- 保存后展示“已分到 XXX 组”结果提示，并提供“立即调整”入口
- 若命中已有组：直接创建卡片并绑定该组
- 若无合适组：Agent 自动创建新组后再创建卡片
- 默认组保留为失败兜底（Agent 或创建新组失败时使用）

### 3.3 复习

**入口流程**：

```
首页或复习 Tab
  → 查看所有卡片组（显示各组待复习数量）
  → 进入某个 Deck
  → 开始该 Deck 的复习 Session
  → Session 结束 → 显示总结页
```

**首页“开始复习”按钮行为**：
- 点击后跳转到“复习 Tab 的 Deck 列表页”（不直接自动进入某个 Deck）
- Deck 默认按“待复习数量”降序展示，并高亮第一个 Deck 作为建议入口

**单张卡片复习流程**：

1. 展示卡片正面（中文）
2. 用户可选输入自己的英文表达（可跳过）
3. 用户若点击"AI 评分"，系统评估输入内容并给出：
   - 得分（0–100）
   - 具体问题说明和改进建议
   - 自动映射到 Anki 四档
4. 展示卡片背面（标准英文答案）
5. 界面始终显示四个手动评级按钮，用户可直接手动评级（不依赖 AI）
6. 若存在 AI 评分，默认高亮其建议档位，但最终以用户点击为准

**AI 评分 → Anki 四档映射**：

| AI 得分 | Anki 档位 | 说明 |
|--------|----------|------|
| 90–100 | Easy | 完全掌握 |
| 70–89 | Good | 基本正确 |
| 50–69 | Hard | 有明显问题 |
| 0–49 | Again | 需要重学 |

**SRS 算法**：v1 直接采用 FSRS 调度（保留 Again / Hard / Good / Easy 评级入口），按记忆保持率目标计算下次复习时间

**FSRS 实现策略（v1）**：
- 优先使用成熟的 Python FSRS 实现库，避免手写公式导致调度偏差
- 前期使用默认参数启动，累积复习历史后再做参数优化
- 若用户复习历史不足，允许回退到保守间隔策略，避免早期调度抖动

**复习口径（对齐 Anki 的可行规则）**：
- 当日应复习集合 = `due_at <= 当前时间` 的卡片
- 当日未完成复习的卡片，不会消失；次日会继续出现在待复习列表（形成积压）
- 次日列表 = 历史积压 + 新到期卡片
- 为避免积压失控，支持“新卡引入上限”与“单日复习上限”配置（v1 可先给默认值）

### 3.4 复习总结页

复习 Session 结束后展示：
- 本次复习卡片数量
- 正确率（Good + Easy 的比例）
- 各档位分布（Again / Hard / Good / Easy）

### 3.5 卡片维护（编辑/删除）

- 用户可编辑卡片正面中文与背面英文
- 编辑卡片后不重置 FSRS 状态（下一次复习时间、稳定度等保持不变）
- 用户可删除卡片，删除前需二次确认

## 八、非业务需求：账号、认证、授权（必须）

- v1 必做账号体系（注册、登录、退出、刷新令牌、找回密码可后置）
- 授权模型：用户仅可访问自己的 Deck/Card 数据（多租户隔离）
- 建议采用 Access Token（短期）+ Refresh Token（长期）模式
- 第三方登录（Google/GitHub）不作为 v1 必选

**认证方案说明（FastAPI 生态）**：
- 当前更稳妥的主流做法是采用 FastAPI 官方 Security 组件 + JWT，而非重依赖单一“全家桶”认证框架
- `fastapi-users` 可用于快速起步，但其官方文档已标注为维护模式（Maintenance Mode），不建议作为长期核心依赖

## 九、注释：Java 开发者转 Python（FastAPI）最需要注意

- 依赖注入方式不同：FastAPI 用函数参数 + `Depends`，不是 Spring 注解扫描；依赖关系是显式声明，便于阅读但需要习惯
- 类型系统用途不同：Python 类型提示主要用于静态检查与编辑器提示，不等于 JVM 的强约束；运行期校验要依赖 Pydantic
- 并发模型不同：I/O 场景优先 `async/await`；阻塞代码会拖慢事件循环，CPU 密集任务需放到后台任务或独立 worker
- 数据访问习惯不同：常用 SQLAlchemy 2.0（显式 Session 生命周期），不要照搬 JPA 的长事务与懒加载心智
- 工程化入口不同：先用 `ruff + mypy + pytest` 建立质量基线，再扩展分层；避免一开始套过重“Java 式”模板

## 十、已确认决策（2026-02-25）

- v1 暂不开放第三方登录（仅邮箱 + 密码 + JWT）
- 首页“开始复习”跳转到 Deck 列表页，不自动进入某个 Deck
- 卡片支持编辑与删除；编辑默认不影响 FSRS 调度
- 有卡片的卡片组不可删除；仅无卡片的非默认组可删除
- 顶部右侧头像采用极简下拉菜单（账号信息占位 + 退出登录）
- 保存卡片统一走 Group Agent：命中已有组则直接创建，未命中则自动创建新组后再创建记录

## 十一、空状态规范（v1）

- 首页空状态：显示“还没有学习卡片”，主按钮“创建第一张卡片”
- 复习页空状态：显示“暂无待复习内容”，主按钮“去记录新内容”
- 卡片组页空状态：始终显示默认组（0 张）与“创建卡片组”入口
- 记录页空状态：保持可输入与可生成英文，不做额外阻断
