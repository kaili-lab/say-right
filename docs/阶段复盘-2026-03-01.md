# Say Right 阶段复盘（2026-03-01）

> 复盘时间：2026-03-01  
> 复盘范围：从 API-001 到 API-015（含前后端联调与本地验证）

## 1) 阶段目标与当前状态

- 目标：完成 v1 主链路（注册登录 -> 记录 -> 存卡 -> 复习 -> 首页统计）并具备可部署基础。
- 当前状态：
  - 主链路接口与前端页面已打通。
  - 数据库 schema 已同步到 Neon。
  - 运行态仓储已从 InMemory 切换为可配置 PostgreSQL（API-015）。
  - AI 相关能力仍是 stub，尚未接入真实 LLM。

## 2) 遇到的问题、根因与改进

### 问题 A：已建表但数据仍不持久化（重启丢数据）

- 现象：Neon 中已有表结构，但 API 重启后业务数据丢失。
- 根因：
  - API-013 只做了 schema 基线，不包含运行态仓储切换。
  - `create_app()` 当时仍实例化 InMemory 仓储。
- 已完成改进：
  - API-015 新增 Postgres 仓储并接入启动装配。
  - 支持通过 `APP_STORAGE_BACKEND` / `DATABASE_URL` 选择运行态存储模式。
- 后续避免方式：
  - 每次涉及“可部署”结论前，必须做一次“重启后数据保留”验证。

### 问题 B：对“任务完成”与“可上线”边界理解不一致

- 现象：任务完成后，体感上以为已满足部署稳定运行。
- 根因：任务拆分里“能力基线”与“生产就绪”是分阶段推进，但未在当下形成统一口径。
- 已完成改进：
  - 已明确 API-013（schema）与 API-015（运行态持久化）是两个不同阶段。
- 后续避免方式：
  - 每次评估上线状态时，按“运行态依赖是否已接入”逐项确认，不只看迁移/建表。

### 问题 C：Python/FastAPI 工程习惯不熟导致启动与环境困惑

- 现象：`venv` 位置、激活方式、命令入口不统一时容易出错。
- 根因：从 Java/Node 迁移到 Python 的第一阶段认知差异。
- 已完成改进：
  - 已补充 `backend/DEVELOPMENT.md`，明确一键命令、自动激活与手动激活方式。
  - 统一推荐 `make -C backend dev` / `make -C backend check`。
- 后续避免方式：
  - 新人或新项目先执行固定流程文档，再进入业务开发。

### 问题 D：AI 相关能力预期偏差（stub vs 真实 LLM）

- 现象：页面“看起来可用”，但实际尚未调用真实模型供应商。
- 根因：早期任务为保证可复现测试，采用了 stub；但“可演示”与“真实 AI”边界需要反复强调。
- 已完成改进：
  - 已在任务与沟通中明确“当前是 stub”。
- 后续避免方式：
  - 后续任务标题与交付说明里持续标注 `stub` / `real provider` 状态。

### 问题 E：工程质量门禁曾出现补齐而非一次性就位

- 现象：`ruff/mypy` 不是在最初第一步就完整执行。
- 根因：项目早期关注点偏功能打通。
- 已完成改进：
  - 已纳入固定流程并统一 `make check` 作为验收入口。
- 后续避免方式：
  - 后续每个后端任务默认以 `make -C backend check` 为完成前置条件。

## 3) 本阶段有效做法（继续保持）

- TDD 小步推进（先红后绿）+ 任务级 output_summary 便于跨会话衔接。
- 接口契约驱动开发，前后端并行时可减少返工。
- 对“前置未就绪”的测试策略做显式声明，避免强行放行。
- 统一命令口径（make）显著降低环境差异导致的问题。

## 4) 下一阶段建议（不立即执行，仅作为规划）

1. 接入 LangChain 统一 LLM 适配层（OpenAI-compatible，baseURL + API Key）。
2. 分阶段替换 3 个 stub：记录生成、复习评分、分组 Agent。
3. 增加 Railway 部署说明（运行命令、环境变量、健康检查、CORS 域名）。

## 5) 一句话结论

- 本阶段最大的认知收获：**“数据库已建表”不等于“应用已持久化运行”**。  
- 现在该问题已在 API-015 完成修正，后续可在此基础上推进真实 LLM 接入与 Railway 部署。
