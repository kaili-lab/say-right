# UI 调整问题与计划 (2026-03-04)

> 来源：用户反馈 8 项 UI 体验问题，经 Playwright 实际截图确认后整理。

---

## 问题 1：手机端无用户入口

**现状**: 桌面端头像菜单使用 `hidden md:block`，手机端完全不可见，用户无法退出登录或管理设置。底部只有 4 个 Tab（首页/记录/复习/卡片组）。

**计划**: 新增第 5 个底部 Tab "我的"（仅手机端），对应 `/me` 路由，页面包含：
- 邮箱（只读显示）
- 昵称（只读，后端目前无修改 API）
- "修改昵称"、"修改密码"按钮标为"即将上线"
- 退出登录

桌面端保持 4 Tab + 头像菜单不变。

**涉及文件**:
- `frontend/src/app/navigation.ts` — 区分 DESKTOP_TABS (4项) / MOBILE_TABS (5项)
- `frontend/src/app/AppShell.tsx` — 底部 nav grid-cols-4 → grid-cols-5，使用 MOBILE_TABS
- `frontend/src/pages/MePage.tsx` — 新页面
- `frontend/src/App.tsx` — 新增 `/me` 路由

**Review 补充（遗漏）**:
- `MePage` 的邮箱/昵称建议直接调用后端 `GET /me`，不要只依赖 localStorage；否则昵称无法准确显示，且会与服务端状态漂移。
- 桌面端虽可保持 4 Tab 不变，但建议把头像菜单里的“账号信息（即将上线）”改为跳转 `/me`，避免个人信息入口维护两套逻辑。
- 需要同步补前端测试（如 `app-shell.test.tsx`）以覆盖新增移动端“我的”入口，避免仅验证旧的 4 个导航断言。

## 问题 2：中文输入框高度固定

**现状**: `RecordPage.tsx` 中文 textarea 固定 `rows={4}` + `resize-y`，初始就占 4 行高度，浪费空间。

**计划**: 改为自动增长：
- 初始 1 行高度
- 随内容输入自动扩展
- 最多 4 行，超出后出现滚动条
- 滚动条不显示上下三角形
- 使用 `useRef` + onChange 动态调整 `textarea.style.height`，配合 `max-height` 限制

**全局滚动条样式** (同时服务于问题 8):
```css
::-webkit-scrollbar { width: 6px; height: 6px; }
::-webkit-scrollbar-track { background: transparent; }
::-webkit-scrollbar-thumb { background: #d6d3d1; border-radius: 3px; }
::-webkit-scrollbar-button { display: none; height: 0; }
* { scrollbar-width: thin; scrollbar-color: #d6d3d1 transparent; }
```

**涉及文件**:
- `frontend/src/pages/RecordPage.tsx` — textarea 改造
- `frontend/src/index.css` — 全局滚动条样式

**Review 补充（遗漏）**:
- 自动高度需要覆盖“内容被清空后回缩到 1 行”的场景，否则会残留上一次输入高度。
- 建议去掉 `resize-y`（改为 `resize-none`），避免用户手动拖拽高度与自动计算逻辑互相冲突。
- 问题 2 和问题 3 都要做自动增长，建议抽一个复用的 textarea 高度同步逻辑，避免两处实现后续偏差。

**现状**: 生成英文后的结果区包含：原文显示、英文 textarea（固定 3 行）、model 和 trace 标签。

**计划**:

- 去掉 model / trace 标签区域

**涉及文件**: `frontend/src/pages/RecordPage.tsx`

**Review 补充（遗漏）**:

- 即使不展示“原文行”，也建议继续保留 `sourceSnapshot` 作为保存入参，避免生成后用户又改中文导致“保存内容”和“生成英文”不一致。

## 问题 4：保存卡片改为手动选分组

**现状**: 保存时调 `saveRecordWithAgent()`，后端 AI agent 自动决定分组，可能产生大量无用分组。

**计划**:
1. 点击"保存卡片" → 弹出分组选择 modal（复用已有 modal 模式）
2. 默认选中 `isDefault=true` 的 deck
3. 用户确认后调新 API `POST /records/save` 直接保存到指定 deck
4. 前端不再调用 `saveRecordWithAgent`（后端 endpoint 保留不删）

**后端新增** `POST /records/save`:
- 请求: `{ source_text, generated_text, deck_id, source_lang, target_lang }`
- 实现: 调 `card_service.create_card()`（由该链路自动刷新 deck 计数）
- 响应: `{ card_id, deck_id, deck_name }`

**涉及文件**:
- `backend/app/record/api.py` — 新增 `POST /records/save`
- `frontend/src/pages/recordApi.ts` — 新增 `saveRecordToDeck()`
- `frontend/src/pages/RecordPage.tsx` — 重构保存流程

**Review 补充（遗漏）**:
- 后端实现里不应再手动调用 `deck_service.update_counts()`：当前 `card_service.create_card()` 底层已自动刷新 deck 计数，重复更新会引入统计漂移风险。
- 需补充 `deck_id` 归属校验（该 deck 必须属于当前用户），并明确错误映射（不存在/无权限建议返回 404）。
- 分组选择默认值要按 `isDefault=true` 命中，不建议仅取列表第一项（列表顺序并非强契约）。
- 该问题会影响现有保存链路测试，需要补 `backend` 集成测试（`POST /records/save`）和 `frontend` 交互测试（从 `/records/save-with-agent` 切换到新接口）。

## 问题 5：保存后英文区域未清空

**现状**: 保存后英文内容保留可编辑，保存按钮仍可点击。

**计划**: 保存成功后：
- 英文内容保留只读（textarea 设 readOnly）
- 显示"已保存到 {deckName}"成功提示
- 保存按钮置灰不可用
- 不隐藏英文结果区

**涉及文件**: `frontend/src/pages/RecordPage.tsx`

**Review 补充（遗漏）**:
- “保存按钮置灰”建议把条件明确为 `saveStatus === "saved"` 后不可再点，避免重复提交同一张卡。
- 保存成功后的 `readOnly` 状态需要在“下一次生成成功”时自动解锁，否则会影响下一轮编辑。
- 保存失败时应保持英文可编辑，以便用户修正后重试。

## 问题 6：卡片组 Tab 卡片内容溢出

**现状**: 手机端 DeckListPage 的 table 中，中文/英文全文展示，长内容占据过多屏幕空间。

**计划**:
- table cell 中 front_text / back_text 加截断（`max-w-[100px] sm:max-w-[200px] truncate`）
- 新增"查看"按钮，点击弹出只读详情 modal
- 详情 modal 展示完整中文、英文、下次复习时间，以及操作入口（编辑/移动/删除）

**涉及文件**: `frontend/src/pages/DeckListPage.tsx`

**Review 补充（遗漏）**:
- 仅加 `truncate` 可能不足以解决手机端可读性，建议同时处理横向溢出（如表格容器 `overflow-x-auto`）或提供移动端卡片式布局。
- 新增“查看”详情后，要避免与现有编辑/移动/删除弹窗形成嵌套弹窗冲突，建议先定义“查看弹窗中如何触发操作弹窗”的交互顺序。
- 该改动会影响既有卡片管理测试（如 `deck-card-management.test.tsx`），需补充“长文本截断 + 查看完整内容”的断言。

## 问题 7：Token 过期无自动刷新，401 不跳转登录

**现状**:
- Access token 15 分钟过期（`ACCESS_TOKEN_EXPIRE_MINUTES = 15`）
- Refresh token 30 天过期（`REFRESH_TOKEN_EXPIRE_DAYS = 30`）
- 后端 `POST /auth/refresh` endpoint 已存在
- 前端无任何 refresh 逻辑，过期后 API 报错"invalid access token"但页面不跳转

**计划**:
1. `authApi.ts` 新增 `refreshAccessToken()`：用 refresh token 调 `POST /auth/refresh`，更新 localStorage 中的 access token
2. `authApi.ts` 新增 `fetchWithAuth(url, init)`：正常请求 → 若 401 → 自动调 refresh → 重试原请求
3. 若 refresh 也 401 → `clearSession()` → `window.location.href = "/auth/login"`
4. 所有 API 模块（homeApi, recordApi, decksApi, reviewApi）的 fetch 改用 `fetchWithAuth`

**涉及文件**:
- `frontend/src/pages/authApi.ts` — 新增 `refreshAccessToken()`, `fetchWithAuth()`
- `frontend/src/pages/homeApi.ts` — 改用 `fetchWithAuth`
- `frontend/src/pages/recordApi.ts` — 改用 `fetchWithAuth`
- `frontend/src/pages/decksApi.ts` — 改用 `fetchWithAuth`
- `frontend/src/pages/reviewApi.ts` — 改用 `fetchWithAuth`

**Review 补充（遗漏）**:
- 后端 `POST /auth/refresh` 读取的是 `Authorization: Bearer <refresh_token>`，这里要在计划里明确，避免误传 body。
- `fetchWithAuth` 需防止无限重试：若 `/auth/refresh` 本身 401，应直接清会话并跳登录，不再递归重试。
- 建议做并发合并（single-flight refresh）：多个请求同时 401 时只发一次 refresh，其余请求复用同一个 refresh Promise。
- 若本地没有 refresh token，应直接 `clearSession()` 并跳登录，不走 refresh 请求。
- 需要新增前端测试覆盖“401->refresh 成功重试”和“refresh 失败->跳登录”两条主路径。

## 问题 8：Web 端页面整体出现滚动条

**现状**: 所有 Tab 页面内容超出 viewport 时，整个页面出现滚动条。手机端这是正常行为，但 Web 端（桌面）各 Tab 应在 viewport 内，只有内容区域超出时才在区域内滚动。

**计划**:
- **Web (md+)**: AppShell root div → `md:h-screen md:flex md:flex-col md:overflow-hidden`
  - header 保持固定
  - `<main>` → `md:flex-1 md:overflow-y-auto`（内容超出时 main 内部滚动）
- **手机端**: 保持正常整页滚动不变
- 滚动条样式在问题 2 中已全局处理（无三角形）

**涉及文件**: `frontend/src/app/AppShell.tsx`

**Review 补充（遗漏）**:
- `md:flex` 布局下要给 `<main>` 补 `md:min-h-0`，否则 `overflow-y-auto` 在某些浏览器不会生效，仍会出现整页滚动。
- 需要确认桌面端 `main` 内滚动后，底部留白（`pb-8`）和移动端底部导航留白（`pb-24`）不会互相影响。
