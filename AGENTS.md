# 项目级 AGENTS 增补规则

本文件是项目级增补规则，与系统/上级规则共同生效，不替代更高优先级指令。

## 语言与沟通

- 所有输出使用中文。

## 任务交付审阅包（强制）

- 每个 task 完成后，必须提供一份“可复查审阅包”，方便非 Python/FastAPI 背景快速验收。
- 审阅包至少包含：
  - 契约对照：接口/状态码/关键字段是否满足 `contract_version`
  - 变更文件清单（带行号引用）
  - 测试证据：执行命令、退出码、关键通过行
  - 范围核对：本 task 做了什么、明确没做什么
  - 已知风险与下一步建议（如有）
- 如任务失败或中断，也要给“失败审阅包”（失败点、原因、建议修复方向）。

## 版本策略

- 默认使用各技术栈的最新稳定版（GA），禁止无理由沿用过时主版本（如在无约束下继续使用 React 18）。
- 若受契约、上游依赖或兼容性约束必须锁定旧版本，需在任务输出中明确说明原因、影响范围与后续升级建议。
- 新增依赖时，优先选择当前生态主流且维护活跃的稳定版本，避免实验版/预发布版进入主分支。

## 学习型注释与文档策略

- 目标：在不制造噪音的前提下，优先帮助初学者快速理解代码结构与职责边界。
- 新增/修改文件时，默认补充：
  - 文件级说明（模块 docstring，说明这个文件做什么、在系统中处于哪一层）
  - 对外类与对外方法的 docstring（说明 WHAT + WHY，必要时补充边界条件）
- 行内注释仍遵循“关键逻辑才写”，避免逐行翻译代码。
- 复杂业务规则、异常映射、兼容性处理必须写中文注释解释原因。

## TDD 执行规则

- 前后端任务默认采用 TDD：先写测试（Red）-> 最小实现（Green）-> 重构（Refactor）。
- 每个任务必须声明测试范围：`test_scope`（`unit` / `integration` / `e2e`）。
- 每个任务必须声明测试命令：`test_commands`，并在完成前实际执行。
- LLM 相关测试必须使用可复现的 stub/fixture；禁止在测试中依赖真实模型调用。

## Subagent 使用边界

- 可使用 Subagent 处理可并行且边界清晰的子任务（如测试脚手架、fixture、文档整理）。
- 主上下文必须负责最终收口：复核变更、运行关键回归、给出最终 DoD 结论。
- 未经主上下文复核的 Subagent 结果，不可视为任务完成。

## Skill 使用规则

- 任务或用户显式写出 `$SkillName` 时，必须优先使用对应 Skill。
- 任务文档存在 `skill_required` 字段时，先读取并执行对应 `SKILL.md`。
- 如果请求的 Skill 不存在或不可读，需明确说明并切换到常规实现流程。

## 任务文档字段约定（建议）

- `skill_required`：本任务必须使用的 Skill 列表。
- `paired_with`：关联的前端/后端任务 ID。
- `contract_version`：接口契约版本号。
- `sync_point`：跨任务联调节点。
- `test_scope`：测试范围（unit/integration/e2e）。
- `test_commands`：执行命令与通过标准。
